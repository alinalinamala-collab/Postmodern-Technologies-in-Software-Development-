# Персональний Git-клієнт (Git UI Client)

Консольний застосунок, який надає зручний інтерактивний інтерфейс (TUI) для виконання базових операцій Git, працюючи як обгортка над системним процесом Git.

##  Мінімальний життєздатний продукт (MVP)
Проєкт реалізує 4 ключові сценарії:
1. Перегляд статусу репозиторію (`git status`).
2. Додавання всіх змін до індексу (`git add .`).
3. Створення коміту з повідомленням (`git commit -m`).
4. Перегляд історії комітів (`git log`).

##  Архітектура та стек технологій
- **Мова:** C# / .NET 8
- **Інтерфейс:** Побудований за допомогою бібліотеки `Spectre.Console` для забезпечення сучасного Terminal UI.
- **Взаємодія з Git:** Здійснюється через клас `System.Diagnostics.Process`. Такий підхід обрано тому, що він дозволяє гнучко викликати системний `git`, перехоплювати стандартний вивід (`stdout`) та помилки (`stderr`) без необхідності підключати важкі сторонні бібліотеки (наприклад, `LibGit2Sharp`). 

Проєкт розбито на два логічні шари (Separation of Concerns):
- **UI Layer (`Program.cs`):** Відповідає виключно за відмальовування меню та взаємодію з користувачем.
- **Service Layer (`Services/GitService.cs`):** Інкапсулює логіку роботи з процесами та імплементує інтерфейс `IGitService`. Це дозволяє легко тестувати логіку або замінити реалізацію в майбутньому (Dependency Injection friendly).

##  Залежності
Для запуску проєкту необхідні:
- Встановлений **.NET 8 SDK**
- Встановлений **Git** (повинен бути доступний у системних змінних оточення)
- NuGet-пакет **Spectre.Console** (встановлюється автоматично при збірці проєкту).

##  Інструкція із запуску

1. Склонуйте або завантажте репозиторій.
2. Відкрийте термінал у кореневій папці проєкту (там, де знаходиться файл `GitUIClient.csproj`).
3. Виконайте команду для запуску:
   ```bash
   dotnet run

## Демонстраційний сценарій

Для перевірки працездатності застосунку виконайте наступні кроки:

 1. Запустіть програму (dotnet run).

 2. Внесіть зміни у будь-який файл проєкту (наприклад, додайте коментар у коді).

 3. Оберіть пункт 1 (Status) — ви побачите змінений файл у списку.

 4. Оберіть пункт 2 (Add .) — зміни будуть додані до staging area.

5. Оберіть пункт 3 (Commit) — введіть тестове повідомлення (наприклад, "Test commit").

6.  Оберіть пункт 4 (Log) — переконайтеся, що ваш коміт з'явився у загальній історії.

7.  Оберіть пункт 0 (Вихід) для завершення роботи.
